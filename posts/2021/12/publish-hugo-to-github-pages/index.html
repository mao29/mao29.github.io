  • У вас еще нет репозитория вида <имя пользователя>.github.io

Чего хотелось достичь

В какой-то момент я наконец нашел мотивацию заняться написанием блога. Выбирая между платформами, а-ля WordPress, и статическими генераторами сайтов решил остановить свой выбор на GitHub Pages. Поскольку для GitHub Pages официальным генератором статических страниц служит Jekyll, написанный на Ruby, а мне не хотелось заниматься установкой Ruby себе на машину, решил в качестве инструмента для создания сайта выбрать HUGO. Хотелось реализовать схему, когда исходные файлы блога находятся в закрытом репозитории, а публикация производится при помощи GitHub Action-а непосредственно в публичный репозиторий для GitHub Pages.

Закрытый репозиторий

Для начала необходимо создать закрытый репозиторий, в котором будут располагаться исходные файлы блога. Сделать это можно следуя официальной инструкции. Лучше создавать пустой репозиторий, не добавляя README-файл, поскольку первым коммитом мы отправим в этот репозиторий свежесозданные файлы HUGO-сайта из локального репозитория и этот README-файл нам не понадобится.

Создание сайта

Для создания сайта можно следовать официальной инструкции с сайта HUGO.

  1. Создать пустой git-репозиторий командой git init
  2. Запустить в папке репозитория команду hugo new site
  3. Выбрать и склонировать тему для сайта в папку themes
  4. Добавить контент сайта Теперь у нас есть локальный git-репозиторий с исходными файлами сайта. Редактировать файлы можно в любом текстовом редакторе или IDE. Я для этих целей использую Visual Studio Code.

Загрузка файлов сайта на GitHub

Чтобы отправить файлы в репозиторий на GitHub, необходимо добавить его в remotes локального гит-репозитория. Для этого выполняем команды

git remote add origin https://github.com/<путь_к_репозиторию>
git branch -M main

После того, как добавили удаленный репозиторий, необходимо сделать начальный коммит. Для этого необходимо сначала добавить все файлы сайта в staging, а затем выполнить команду коммита, добавив комментарий.

git add *
git commit -m "Начальный коммит"

Чтобы не выполнять все команды в терминале, можно воспользоваться инструментами работы с гитом в выбранном вами редакторе. Как правило редакторы и IDE позволяют в WYSIWYG режиме добавлять файлы, осуществлять коммит и отправку изменений в удаленный репозиторий.

После того, как сделан начальный коммит, можно отправить его в удаленный репозиторий командой git push -u origin main. В результате на странице репозитория на GitHub должны появиться исходные файлы сайта. Можно приступать к настройке публикации.

Публикация сайта в GitHub Pages

Сначала необходимо создать репозиторий с именем <user>.github.io. Такое имя репозитория по умолчанию используется для хранения файлов сайта, развернутого с помощью GitHub Pages.

Для сборки сайта официальная документация HUGO предлагает использовать GitHub Action. Если изучить его описание внимательно, он состоит из следующих шагов:

  1. Получаем файлы из репозитория
  2. Устанавливаем в образ для сборки HUGO
  3. Выполняем сборку сайта
  4. Публикуем результаты сборки в ветку gh-pages данного репозитория Такая схема работы подходит и для нашего случая за исключением последнего шага, так как на этом шаге нам необходимо отправить результаты сборки в отдельный репозиторий для GitHub Pages. К счастью, action, используемый на 4 шаге, поддерживает публикацию результатов во внешний репозиторий. Для этого необходимо изменить настройки последнего шага на следующие:
- name: Deploy
  uses: peaceiris/actions-gh-pages@v3
  if: github.ref == 'refs/heads/main'
  with:
    deploy_key: ${{ secrets.DEPLOY_KEY }}
    publish_dir: ./public
    external_repository: <user>/<user>.github.io
    publish_branch: main

Здесь мы указали в external_repository ссылку на наш репозиторий для GitHub Pages, изменили ветку, в которую отправляются результаты сборки на main (по умолчанию используется gh-pages), а также указали ключ, используемый для отправки файлов во внешний репозиторий deploy_key.

Каким образом сформировать и настроить ключ публикации описано в инструкции к action-у. В результате, после коммита и отправки измененного файла с описанием action-а публикации, этот action должен выполниться и файлы собранного сайта должны появиться в публичном репозитории для GitHub Pages в ветке main. Теперь необходимо настроить доступность сайта по адресу https://<user>.github.io.

Настройка доступности сайта

Необходимо изменить настройку baseURL в config.toml на адрес https://<user>.github.io, закоммитить и отправить данные изменения. После того, как обновленные файлы сайта опубликуются в репозиторий с помощью action-а из предыдущего шага, настраиваем GitHub Pages.

  1. Переходим на страницу Settings репозитория <user>.github.io во вкладку Pages
  2. В параметре Source выбираем ветку main и директорию / (root) в качестве источника файлов для сайта.

После осуществления данных настроек сайт должен быть доступен по адресу https://<user>.github.io.

В дальнейшем можно настроить для сайта собственное доменное имя, но это тема уже отдельной статьи.
